## 1. 复盘目的

本文档用于沉淀当前项目阶段的开发经验、工程收获、已验证结果、暴露问题与后续建议，作为后续迭代、交接、版本收口和团队协作的统一参考。

本次复盘不是站在“功能是否写完”的角度总结，而是站在“项目是否已经具备持续迭代能力”的角度进行回顾。

## 2. 当前阶段结论

当前项目已经从“玩法原型”推进到“可持续迭代的玩法版本”阶段。

核心判断依据如下：

- 主流程已经形成闭环，涵盖启动加载、选关、地图翻页、进入关卡、战斗内 HUD、胜负结算、返回选关。
- 玩法运行时不再依赖单一大脚本，而是逐步拆分为发射、瞄准、网格、匹配、断层、掉落、收集等独立系统。
- 关卡内容已经完成数据化表达，并且具备了特殊球、障碍球等可扩展结构。
- 项目已经配套内容校验、难度曲线分析、瞄准参数校验、射击回归等工具脚本，不再完全依赖人工肉眼验证。
- 音频、体力、进度、设置页、资源预热等外围能力已经开始纳入主工程，而不是停留在“临时功能”状态。

换句话说，当前版本的价值不只是“可以玩”，而是已经具备“继续扩关卡、扩机制、做收口”的工程基础。

## 3. 本轮最有价值的工程收获

## 3.1 玩法系统边界被明确拆开

本项目最值得肯定的一点，是没有把所有玩法逻辑继续堆在一个总控脚本里，而是逐步拆成了职责清晰的运行时系统。

当前主链路大致如下：

- `GameBootstrap` 负责启动流程、UI 串联、资源初始化和运行环境收口。
- `GameManager` 负责玩法回合推进、分数结算、危险线、胜负判定和跨系统协调。
- `ShooterController` 负责炮台当前球/下一球、技能球库存和瞄准状态。
- `TrajectoryPredictor` 负责射线预测、墙反弹和落点推导。
- `BubbleGrid` 负责六边形网格、落点候选、特殊实体映射与网格坐标逻辑。
- `MatchSystem` 负责同色连通匹配。
- `SupportSystem` 负责断层检测。
- `FallingMarbleSystem` 负责掉落球的动态表现、缸口反弹、收集区与失效区结算。
- `JarCollectorSystem` 负责收集统计与目标进度。
- `LevelRenderer` 负责战斗场景和反馈表现。

这类拆分的直接收益有三点：

- 新玩法接入成本下降，彩虹球、炸裂球、石头球、冰球这类机制不需要整体推翻已有逻辑。
- 调试范围可控，问题能更快定位到具体系统而不是整条玩法链。
- 后续如果要补自动化验证，也更容易围绕系统边界设计脚本。

## 3.2 关卡内容真正进入数据驱动阶段

当前关卡已经不再只是简单的 `layout` 文本，而是逐步形成了可扩展的数据结构。

当前设计中比较成熟的做法包括：

- `layout` 继续只承担普通球排布表达，保持基础规则清晰。
- `specialEntities` 单独描述技能球和障碍球，避免把复杂语义塞进单字符编码。
- `jarRules`、`spawnWeights`、`winConditions`、`bonusObjectives`、`difficultyScore` 等字段让关卡不只描述“长什么样”，还描述“怎么打”和“想让玩家感受到什么”。

这意味着后续策划扩内容时，工程可以优先扩配置规范和校验工具，而不是每次都从玩法代码里加分支。

## 3.3 手感参数被配置化，而不是散落在逻辑里

当前项目对“手感”的管理已经开始工程化。

主要表现为：

- 棋盘、球半径、危险线、炮台原点、缸体尺寸等基础布局统一集中在 `BoardLayout`。
- 瞄准容差、碰撞半径、通道辅助半径、槽位吸附紧度等参数统一集中在 `AimTuningProfiles`。
- 分数热度和分档逻辑在 `GameManager` 中有统一口径，不再由各个行为随意加分。
- 掉落反弹、吸附、寿命、碰撞衰减等参数在 `FallingMarbleSystem` 中成组管理。

这类配置化的意义很大：

- 手感调优可以集中进行，不必反复翻业务逻辑。
- 难度分层可以通过参数组合体现，而不是通过“隐性规则”制造不透明难度。
- 内容团队和开发团队可以围绕统一参数体系协同。

## 3.4 项目开始具备“内容生产闭环”

这是本项目从原型走向产品化最重要的一步。

当前已经存在的脚本能力包括：

- `tools/validate-level-content.js`：校验关卡结构、字段和跨字段一致性。
- `tools/analyze-level-curve.js`：分析难度曲线和阶段均值。
- `tools/validate-aim-profiles.js`：校验瞄准参数是否在合理范围内。
- `tools/validate-shot-regression.js`：验证关键射击案例的结果稳定性。
- `tools/sync-level-configs.js`：同步运行目录和编辑目录中的关卡文件。

这说明项目已经不再只依赖“进编辑器跑一下看看”，而是开始建立基础的工程回归机制。

对玩法项目而言，这一点非常关键。因为真正拖垮版本效率的，通常不是单个功能难，而是内容越多以后每次改动都没有可靠兜底。

## 3.5 主流程与外围系统被纳入统一工程，而不是零散补丁

当前项目没有把“设置页、体力、进度、启动加载、资源预热、选关页状态”当成后补内容，而是纳入了启动层统一管理。

已经落地的方向包括：

- 关卡进度存档与星级记录。
- 玩家资源与每日体力刷新。
- 启动加载页和预加载。
- 选关页地图翻页与状态控制。
- 设置页的音乐、音效、震动与音量调节。
- 启动调试参数和测试入口。

这类能力虽然不如玩法机制“显眼”，但对版本稳定性和后续收口非常关键。

## 3.6 文档化意识明显提升

当前仓库中已经存在以下文档资产：

- 游戏设计文档
- 关卡配置规范
- 特殊实体配置规范
- 技能球说明
- 开发计划
- 当前交接文档

这说明项目已经开始把“设计规则”“字段约束”“迭代结论”沉淀下来，而不是只留在聊天记录或个人记忆里。

这对后续多人协作、版本切换和长期维护非常重要。

## 4. 本轮开发给出的关键方法论

结合本项目现状，可以沉淀出以下几条方法论：

### 4.1 玩法项目一定要先拆系统，再扩机制

如果不先拆系统边界，后续每增加一个机制都会变成对总控脚本的持续侵蚀。短期看开发更快，长期看调试和回归成本会指数上升。

本项目已经证明，把发射、预测、匹配、断层、掉落、收集分开，是值得的。

### 4.2 内容字段必须先规范，再规模化扩关

只要开始做 20 关以上的内容，纯靠“约定俗成”就会迅速失控。必须先明确字段约束、命名规则、目标口径和目录权威性，再继续扩内容。

本项目中 `specialEntities` 的做法是正向案例。

### 4.3 手感参数必须集中，否则无法稳定调优

瞄准、反弹、危险线、缸体吸附、发射速度这类参数如果分散在多个脚本里，就会造成调一个地方、坏另一个地方。

本项目把这部分往集中配置推进，是正确方向。

### 4.4 工具链要尽早建立，而不是临发布前补

内容越多，越不能只依赖人工验证。特别是玩法项目，一旦没有校验脚本和回归样例，后续每次改动都会越来越慢、越来越不敢改。

### 4.5 UI 流程和玩法流程都必须有“收口层”

玩法系统不能直接承载所有 UI 流转，UI 流程也不能反过来侵入具体玩法细节。

当前项目用 `GameBootstrap` 做主流程收口，是比较合理的工程选择。

### 4.6 Web 音频和移动端体验要单独看作高风险项

这类问题通常不是逻辑错误，而是平台策略差异。单机浏览器可用，并不等于移动浏览器真机可用。

因此这类问题必须建立真机验证矩阵，而不能只在桌面环境完成验收。

## 6. 当前暴露出的工程问题

### 6.1 21~30 关缺少 `targetScore`

本地校验结果显示，`level_021.json` 到 `level_030.json` 缺少 `targetScore` 字段，导致内容校验失败。

这说明新增关卡已经进入运行目录，但字段规范还没有和前 20 关保持完全一致。

影响如下：

- 星级目标和分数热度带会失去统一口径。
- 内容校验脚本无法通过。
- 后续如果要做星级回归和数值对照，会缺少明确基线。

### 6.2 内容校验脚本仍保留 “20 关 MVP” 旧约束

当前项目实际已有 30 个关卡文件，但 `validate-level-content.js` 仍然保留 “Expected 20 levels for MVP” 的旧判断。

这属于典型的“内容规模已经扩了，但工具脚本没有同步更新”问题。

影响如下：

- 脚本失败中混入了“真实结构问题”和“过期阶段性约束”两类信息。
- 校验结果噪声变大，不利于后续持续使用该脚本。

### 6.3 难度曲线脚本和当前难度标签口径不完全一致

当前难度分析脚本使用 `tutorial/easy/normal/hard/expert` 五档聚合，但现有关卡中大量使用 `advanced`。

这导致：

- `easy` 和 `normal` 平均值为 0。
- 阶段均值告警带有明显的口径失真。
- 曲线脚本输出的告警不完全等于真实内容问题。

这说明脚本与当前关卡分层策略之间还存在脱节。

### 6.4 Web 真机音频仍是当前版本首要风险

从现有代码和交接记录看，WebAudio 解锁链路已经做了多轮加固，但这一类问题仍然需要 Android Chrome 和 iOS Safari 真机复测才能真正收口。

这不是代码层面的否定，而是该问题天然具有平台不确定性。

## 7. 对当前项目的整体评价

如果从“工程基础”角度评价，这个项目当前已经做对了很多关键事情：

- 没有继续让玩法逻辑失控堆积。
- 没有把关卡扩展建立在隐式约定上。
- 没有把手感调试完全依赖人工经验。
- 没有等到后期才开始补工具和文档。

如果从“版本收口”角度评价，目前还差最后几项关键工作：

- 对新增关卡字段规范补齐。
- 对校验脚本与当前内容规模做一次同步升级。
- 对难度分层规则和分析脚本口径统一。
- 对移动端 Web 音频做真机矩阵验证。

因此，当前项目最准确的定位不是“已彻底完成”，而是“已经搭好了继续往发布版本推进的正确工程骨架”。

## 8. 下一阶段建议

建议按以下优先级推进。

### P0：修正内容规范与校验脚本

- 为 `level_021` 到 `level_030` 补齐 `targetScore`。
- 更新 `validate-level-content.js` 中关于关卡总数的阶段性约束。
- 明确当前版本的目标内容规模，避免脚本继续使用旧的 MVP 假设。

### P1：统一难度标签与曲线分析口径

- 明确 `advanced` 是否仍作为正式难度标签保留。
- 如果保留，则同步更新 `analyze-level-curve.js` 的分层聚合逻辑。
- 如果不保留，则统一迁移为现有标准难度档位。

### P1：完成真机音频验证矩阵

- 验证 Android Chrome 首次触摸后 BGM 与 SFX 的恢复情况。
- 验证 iOS Safari 的首次交互、切后台返回、静音模式等场景。
- 固化最小复现步骤和通过标准。

### P2：形成发布前 RC 检查清单

- 启动加载
- 选关页进入
- 地图翻页
- 进入战斗
- 掉落入缸
- 胜利结算
- 失败结算
- 返回选关
- 设置页开关与音量控制
- 体力扣减与刷新

### P2：继续保持“文档 + 工具 + 配置”同步推进

- 每次扩字段，先补规范。
- 每次扩关卡，先跑校验。
- 每次调手感，保留参数口径。
- 每次阶段收口，更新交接文档和复盘文档。
